# Migration Guide — M2M OAuth2 client_credentials

This document covers the changes made on **2026-05-21**, following migration `07.MIGRATIONS-06052026.md`. The Access Management synchronization layer migrates from a static M2M token to OAuth2 `client_credentials`, and several dead config fields are removed.

## What changed

1. **OAuth2 `client_credentials` replaces `IGRP_M2M_TOKEN`.** The framework now obtains a bearer token from `POST {IGRP_ACCESS_MANAGEMENT_API}/oauth2/token` using a Basic-auth header derived from `(IGRP_M2M_CLIENT_ID, IGRP_M2M_CLIENT_SECRET)`. The token is cached per-process and refreshed transparently on expiry.
2. **`IGRP_M2M_SERVICE_ID` is renamed to `IGRP_SERVICE_ID`.** The value is *not* a credential — it's the service identity sent as the `X-Machine-Service-ID` header and used as the resource `name` on the Access Management server. The new prefix discipline: `IGRP_M2M_*` for auth credentials only; service identity drops the `M2M` prefix.
3. **`IGRP_M2M_SCOPE` is added** as an optional OAuth2 scope override. Leave blank to use the AM authorization server's default scope for this client.
4. **`IGRP_SYNC_ON_CODE_MENUS` is removed.** The variable had no effect — `IGRP_SYNC_ACCESS` gates all sync phases, including on-code menu sync. The field was always dead.
5. **`apiManagementConfig.m2mToken`, `m2mServiceId`, and `syncOnCodeMenus` are removed** from the type at `@igrp/framework-next-types`. Templates that read them fail to type-check; the error guides the rename.
6. **Validation now happens at render time.** When `IGRP_SYNC_ACCESS=true` and `IGRP_PREVIEW_MODE=false`, the framework validates every required env var inside `IGRPRootLayout` and throws `IgrpConfigError('IGRP_ACCESS_MANAGEMENT_CONFIG_MISSING')`. The error reaches `app/global-error.tsx` rather than disappearing into a `console.error` from a post-stream `after()`. Misconfigured environments fail fast and visibly on the first request.

## Why

- The static M2M token was a long-lived secret that couldn't be rotated without redeploying every consumer. OAuth2 `client_credentials` shifts the long-lived secret to the `client_secret` and gives the AM server fine-grained control over token lifetime, scope, and revocation.
- Three separate `AccessManagementClient` instances were being created per startup (one per sync phase), each with its own token cache. The new shared client + token cache reduces auth round-trips to one per process lifetime instead of three per request that triggered sync.
- `syncOnCodeMenus` was set by the template and never read by the framework — a textbook case of dead configuration that misleads operators ("I set the flag, why doesn't it do anything?").

## Manual review steps

This migration replaces `.env.example` and `src/igrp.template.config.ts` wholesale. After applying:

1. **Diff your local `.env.example`** if you customized it. The migration's `.env.example` is the new canonical reference; cross-check any keys you added that aren't in the new file.
2. **Update your real `.env`**:
   - Remove `IGRP_M2M_TOKEN`.
   - Remove `IGRP_SYNC_ON_CODE_MENUS`.
   - Rename `IGRP_M2M_SERVICE_ID` → `IGRP_SERVICE_ID`.
   - Add `IGRP_M2M_CLIENT_ID` and `IGRP_M2M_CLIENT_SECRET` (request them from your Access Management admin).
   - Optionally add `IGRP_M2M_SCOPE` if your AM admin issues scoped tokens.
3. **Register the OAuth2 M2M client** on your IGRP Access Management server, if not already done. You'll need the issued `client_id` + `client_secret`.
4. **Restart the dev server.** A successful boot with `IGRP_SYNC_ACCESS=true` shows `Access Management sync completed.` in the server console.

## Escape hatch

If you can't migrate to OAuth2 immediately, the downstream library (`@igrp/platform-access-management-client-ts`) still supports the legacy static `token` via `M2MClientConfig.token` (marked `@deprecated`). To use it, pin `@igrp/framework-next` to its pre-migration-08 version (`0.1.0-beta.136` or earlier) until you can migrate.
