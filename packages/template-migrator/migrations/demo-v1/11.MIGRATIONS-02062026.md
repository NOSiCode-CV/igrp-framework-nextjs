# Migration Guide — callbackUrl hardening + AppError error-copy

This document, following migration `10.MIGRATIONS-02062026.md`, back-fills template changes that landed in the reference template but were never captured by a migration. It bundles two coherent features plus one config-surface addition.

## What changed

### A. callbackUrl hardening (open-redirect + login-loop prevention)

Four files cooperate to make `callbackUrl` safe and basePath-aware end to end. The shared `sanitizeCallbackUrl` helper they all rely on already ships in `src/lib/utils.ts` (migration 10).

1. **`src/middleware.ts`** — injects an `x-current-path` request header, builds the unauthenticated redirect to a basePath-aware `/login` with a `sanitizeCallbackUrl`-validated `callbackUrl`, and bypasses static asset paths via an extension check. Replaces the previous `auth.getLoginRedirectUrl` call with explicit local logic.
2. **`src/lib/auth.ts`** — adds a NextAuth `redirect` callback that rejects off-origin URLs, strips `/login`/`/logout` targets (the login-loop guard), and derives the app base URL by removing the `/api/auth` suffix from `NEXTAUTH_URL`. Sets `pages.signIn: "/login"`, adds dev-only session diagnostics, and imports the provider factory from the `@igrp/framework-next-auth/providers` subpath.
3. **`src/lib/dal.ts`** — on a missing session, reads the `x-current-path` header (the contract emitted by middleware) and redirects to `/login?callbackUrl=<encoded>` instead of a bare `/login`.
4. **`src/app/(auth)/login/page.tsx`** — runs the incoming `callbackUrl` search param through `sanitizeCallbackUrl` (falling back to `/`) before handing it to `IGRPAuthForm`, so a crafted or nested value can't drive an open redirect or a `?callbackUrl=…?callbackUrl=…` loop.

### B. AppError digest-encoded error copy

5. **`src/lib/errors.ts`** (new file) — re-exports `parsePublicDigest`, `AppError`, and `logger` from `@igrp/framework-next` so the template imports them from a stable `@/lib/errors` path, and defines the template's local `EnvValidationError` / `AuthError` classes.
6. **`src/config/error-messages.ts`** — `resolveErrorCopy` now calls `parsePublicDigest(error.digest)` to surface a public message encoded in an `AppError`'s digest, instead of always falling back to generic copy when no framework `code` is present.
7. **`src/app/global-error.tsx`** — passes `resolveCopy={resolveErrorCopy}` into `IGRPGlobalError`, so the global boundary actually renders the resolved (code/digest-aware) message.

### C. Config surface

8. **`src/lib/config/get-pkj.ts`** — adds a `slug: process.env.NEXT_PUBLIC_BASE_PATH || ""` field to the returned app-info object.

## Why

- The callbackUrl path was the source of two real defects: an open-redirect vector (scheme-relative / absolute targets) and a login loop where a successful sign-in bounced back to `/login` with an ever-growing nested `callbackUrl` chain. Centralising on `sanitizeCallbackUrl` and the `x-current-path` contract closes both.
- Error boundaries previously discarded the resolved copy on the global path, so server-thrown `AppError`s showed generic text. Wiring `parsePublicDigest` + `resolveCopy` lets a server error carry a safe, public message to the client boundary via its digest.

## Manual review steps

This migration replaces `src/middleware.ts`, `src/lib/auth.ts`, `src/lib/dal.ts`, `src/app/(auth)/login/page.tsx`, `src/config/error-messages.ts`, `src/app/global-error.tsx`, and `src/lib/config/get-pkj.ts` wholesale, and **creates** `src/lib/errors.ts`.

1. **If `src/lib/errors.ts` already exists** in your app (you added it manually), this migration overwrites it with the template version. Diff it first if you had local additions — `rollback 11-callbackurl-hardening-and-error-copy` restores your prior copy.
2. **If you customized `middleware.ts` or `lib/auth.ts`** (e.g. extra public routes, custom providers), re-apply those edits on top of the new files. The new middleware keys public-path detection off path/extension checks — fold any custom allowlist back in.
3. **Confirm `NEXTAUTH_URL` includes the basePath and `/api/auth`.** The redirect callback derives the app base URL by stripping the `/api/auth` suffix; a malformed `NEXTAUTH_URL` will break post-login redirects.
4. **Restart the dev server** and verify: an unauthenticated request redirects to `/login?callbackUrl=…` (sanitized, single-level), a successful login lands on the originating page (never back on `/login`), and a server-thrown `AppError` shows its public message on the error screen.

## Caution

The middleware and auth-redirect logic is security-sensitive. If your deployment fronts the app with a proxy that rewrites paths, validate that `x-current-path` and the basePath-prefixed redirects resolve correctly behind it before shipping to production.
