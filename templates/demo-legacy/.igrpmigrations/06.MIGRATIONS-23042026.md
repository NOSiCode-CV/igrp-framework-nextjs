# Migration Guide - App Router Error Handling Overhaul

This document covers the changes made on **2026-04-23**, following migration `05.MIGRATIONS-23042026.md`. Two related improvements:

1. A typed error hierarchy added to `@igrp/framework-next` and a new segment-level error component added to `@igrp/framework-next-ui`.
2. A full pass over the template's App Router error-boundary coverage (`global-error.tsx`, `error.tsx` per segment, `not-found.tsx`) so every server throw lands somewhere sensible — in both development and production — with localized copy and a single observability hook.

## Problem

Running the template with invalid or missing config produced an unhelpful error screen: a raw `Error('[ERROR_IGRP_PREVIEW_MODE]: ...')` thrown from inside `IGRPLayout`, a `(igrp)/error.tsx` boundary that used a full-page component meant for root-segment errors, no `global-error.tsx` at all, and no boundary for anything rendered under `(auth)/*` or the root. Production surfaces were worse — Next.js redacts `error.message`, so the user sees only "An error occurred" with no branching possible on the raw string.

## Root Cause

Three structural gaps:

1. **Framework throws were untyped.** Every server-side failure path in `@igrp/framework-next` threw a bare `new Error(...)` with a Portuguese message prefix. In production Next redacts `error.message`; without `error.name` / `code` metadata, client boundaries had no way to discriminate failure modes.
2. **The `igrp-layout.tsx` throw fired from inside a layout.** Errors thrown while rendering a layout bubble to the *parent* segment's boundary, not to sibling `error.tsx` files. With no `app/error.tsx` and no `app/global-error.tsx`, the throw hit Next's built-in fallback page.
3. **The one existing `error.tsx` used the wrong component.** `app/(igrp)/error.tsx` rendered `IGRPGlobalError`, which is designed for `global-error.tsx` (no layout chrome around it) — it fought the layout chrome it was mounted inside.

## Package Changes — `@igrp/framework-next`

### 1. Typed error hierarchy

New file: `src/errors.ts`. New subpath export: `@igrp/framework-next/errors`. The same symbols are also re-exported from the main barrel.

```typescript
import {
  IgrpError,
  IgrpConfigError,
  IgrpAuthConfigError,
  IgrpLayoutDataError,
  isIgrpError,
  type IgrpErrorCode,
  type IgrpErrorContext,
  type IgrpErrorShape,
} from '@igrp/framework-next/errors';

throw new IgrpConfigError('IGRP_CONFIG_NOT_INITIALIZED', 'Detalhe opcional');
```

Each subclass sets `this.name` so production serialization across the server→client boundary preserves identity, and carries:

- `code: string` — stable machine-readable tag (`IGRP_ACCESS_MANAGEMENT_CONFIG_MISSING`, `IGRP_APP_CODE_MISSING`, `IGRP_APP_HOME_SLUG_INVALID`, etc.).
- `context?: Record<string, string | number | boolean | null>` — serializable trace data.

`isIgrpError(error)` is a structural guard that works in both dev (real `instanceof`) and prod (plain serialized object shaped like `IgrpError`).

### 2. Validation moved into `igrpBuildConfig`

`src/lib/build.ts` now asserts on the config it receives:

- `IgrpConfigError('IGRP_CONFIG_NOT_INITIALIZED')` when `config` is null/undefined.
- `IgrpConfigError('IGRP_ACCESS_MANAGEMENT_CONFIG_MISSING')` when `previewMode` is false and `apiManagementConfig.baseUrl` is missing.

Because `igrpBuildConfig` runs at the **root segment** (the template calls `createConfig` from `app/layout.tsx`), any throw here bubbles directly to `app/global-error.tsx` — the only boundary that can render when the root layout itself fails.

### 3. `src/layouts/igrp-layout.tsx` — duplicate check removed

The access-management validation inside `IGRPLayout` was removed; the remaining code simply calls `igrpSetAccessClientConfig({...})` inside the same `!previewMode` branch, under the invariant that `apiManagementConfig.baseUrl` is non-empty by the time execution reaches the layout (enforced upstream in `igrpBuildConfig`).

### 4. `src/hooks/use-layout.ts` — typed layout-data error

`throw new Error('[igrp-layout]: Código da aplicação não encontrada.')` became `throw new IgrpLayoutDataError('IGRP_APP_CODE_MISSING', ...)`.

## Package Changes — `@igrp/framework-next-ui`

### 5. New `IGRPSegmentError` component

New file: `src/components/errors/segment.tsx`. Exported from the package barrel alongside the existing `IGRPGlobalError`.

Designed for **segment-level** `error.tsx` — sits inside existing layout chrome, shorter height than `IGRPGlobalError`, offers **reset** + **go home** actions, uses semantic design-system tokens, and accepts a `resolveCopy(error) → { title, description }` injection so templates can translate copy by error `code`.

`IGRPGlobalError` stays as-is for root-level `global-error.tsx`.

## Template Changes — `templates/demo-legacy`

### 6. Single reporting hook — `src/lib/report-error.ts`

One function: `reportError(error, context?)`. In development it `console.error`s with segment + error + context. In production it emits a serializable payload (`name`, `message`, `digest`, `code`, `context`) that's ready to be wired into Sentry / OTLP later. Never throws.

### 7. i18n copy map — `src/config/error-messages.ts`

Portuguese copy keyed by framework `IgrpError.code`:

```typescript
export function resolveErrorCopy(error: unknown): ErrorCopy {
  // Looks up `error.code` in the map, falls back to generic copy.
}
```

Extensible to a real i18n library later without changing call sites.

### 8. Error boundaries — full App Router coverage

```
src/app/
  global-error.tsx       ← NEW   (ships own <html>/<body>; renders IGRPGlobalError)
  error.tsx              ← NEW   (root-level segment boundary; IGRPSegmentError)
  not-found.tsx          ← kept
  (auth)/
    error.tsx            ← NEW   (auth-specific fallback copy)
  (igrp)/
    error.tsx            ← rewritten to use IGRPSegmentError (was IGRPGlobalError)
```

Every boundary calls `reportError(error, { segment })` inside `useEffect` and passes `resolveErrorCopy` into the component so the UI shows localized strings keyed on `IgrpError.code` rather than the raw `error.message` (which is redacted in production).

### 9. Error leak cleanup

- **`src/lib/auth.ts`** — `serverSession()` no longer swallows every error into `null`. Typed `IgrpError` instances now propagate to the nearest boundary via `if (isIgrpError(error)) throw error`. Only the "no session" branch returns `null`, with `reportError` called first. Hard-fails in production when `NEXTAUTH_SECRET` is missing.
- **`src/app/(igrp)/page.tsx`** — the raw `throw new Error("Root redirect must be a valid path")` for an invalid `NEXT_PUBLIC_IGRP_APP_HOME_SLUG` became `throw new IgrpConfigError('IGRP_APP_HOME_SLUG_INVALID', ...)`.
- **`src/app/(auth)/logout/page.tsx`** — `signOut()` wrapped in try/catch with `reportError` on failure, a `router.replace('/')` fallback on completion, plus a 3-second safety timeout so the page never hangs if the async chain stalls.

### 10. Redundant catch-all 404

`src/app/[...not-found]/page.tsx` is marked deprecated in-file. It is redundant with `src/app/not-found.tsx` and should be deleted:

```bash
rm -rf templates/demo-legacy/src/app/\[...not-found\]
```

Pending this manual step because the sandbox running Claude cannot delete workspace files. No behavior changes until the directory is removed.

## Migration Steps

### 1. Pull the updated packages

From repo root:

```bash
pnpm install
pnpm build:framework
```

`pnpm build:framework` rebuilds the ordered chain so the new `IgrpError` symbols and `IGRPSegmentError` component are available to the template.

### 2. Add the template-side helpers

Create `src/lib/report-error.ts` and `src/config/error-messages.ts` from this migration. These are the single points of coupling to observability and i18n respectively — keep them small and swap the internals later without touching call sites.

### 3. Add the App Router boundaries

Create `src/app/global-error.tsx`, `src/app/error.tsx`, and `src/app/(auth)/error.tsx`. Rewrite `src/app/(igrp)/error.tsx` to use `IGRPSegmentError` with `resolveErrorCopy`. Every boundary calls `reportError(error, { segment })`.

### 4. Clean template error leaks

- `src/lib/auth.ts`: add `if (isIgrpError(error)) throw error;` before the silent `return null` fallback. Call `reportError` on the non-typed branch. Promote the `NEXTAUTH_SECRET` check to a hard throw in production.
- `src/app/(igrp)/page.tsx`: replace the raw `throw new Error(...)` for the invalid home slug with `throw new IgrpConfigError('IGRP_APP_HOME_SLUG_INVALID', ..., { receivedValue: root })`.
- `src/app/(auth)/logout/page.tsx`: wrap `signOut()` in try/catch, call `reportError` on failure, redirect to `/` unconditionally in `finally`, and add a 3 s safety `setTimeout`.

### 5. Delete the deprecated catch-all

```bash
rm -rf src/app/\[...not-found\]
```

### 6. (No other changes required)

`package.json`, `src/middleware.ts`, `src/lib/utils.ts`, `src/app/layout.tsx`, and `src/app/(igrp)/layout.tsx` are unchanged by this migration. Preview mode + auth bypass from migration `05` continue to work.

## Verification

Run with the template:

```bash
pnpm --filter @igrp/framework-next-template dev
```

Expected behavior per scenario:

| Scenario | Trigger | Expected |
|---|---|---|
| Baseline boot | default env | App renders with mock/real session, no errors logged |
| Config missing | `IGRP_PREVIEW_MODE=false`, unset `IGRP_ACCESS_MANAGEMENT_API` | `global-error.tsx` catches `IgrpConfigError('IGRP_ACCESS_MANAGEMENT_CONFIG_MISSING')`; UI shows the mapped Portuguese copy; console/observability receives code + digest |
| Invalid home slug | `NEXT_PUBLIC_IGRP_APP_HOME_SLUG=foo` | `(igrp)/error.tsx` catches `IgrpConfigError('IGRP_APP_HOME_SLUG_INVALID')`; layout chrome remains; reset button re-renders segment after env fix |
| Auth bypass (none) | `AUTH_PROVIDER=none`, `IGRP_PREVIEW_MODE=false` | From migration `05`: app renders with mock session; `/login`, `/logout`, `/api/auth/*` all 302 to `/` |
| Auth bypass (preview) | `AUTH_PROVIDER=keycloak`, `IGRP_PREVIEW_MODE=true` | Same as above |
| Unknown URL | visit `/completely-random-path` | `app/not-found.tsx` renders `IGRPTemplateNotFound` |
| Real auth flow | valid Keycloak env, preview off | Sign-in + refresh + sign-out unchanged from migration `04`; any provider config failure surfaces through `(auth)/error.tsx` instead of an opaque NextAuth error page |
| Logout when auth is disabled | bypass mode + visit `/logout` | Middleware 302s to `/`; if accessed by other means, `signOut()` failure is reported and the page redirects within 3 s |
| Production build | `pnpm build:demo && pnpm start:demo` | `error.message` redacted by Next; UI still shows meaningful copy because `resolveErrorCopy` keys off `error.code` (preserved across server→client edge via `error.name`) |

## Open Items

- **Changesets.** Items 1–5 in this migration change the public API of `@igrp/framework-next` and `@igrp/framework-next-ui`. Per the repo's `hard-rules.md`, a changeset entry is required for each before the next release. Suggested summary: *"Typed IgrpError hierarchy + ./errors entry point; IGRPSegmentError for segment-level boundaries; access-management config validation moved to igrpBuildConfig."*
- **Observability wiring.** `reportError` currently logs to the console in both environments. Replace the production branch with your Sentry / OTLP / custom endpoint when the team's observability stack is chosen. `digest` is the correlation ID that maps to the server-side log entry.
- **`IGRPGlobalError` component copy.** It currently hardcodes Portuguese strings and renders `error.message` directly (which is redacted in production). A follow-up pass should adopt the same `resolveCopy` injection pattern as `IGRPSegmentError` so the two components stay consistent.
- **`templates/demo`.** Intentionally out of scope per the user's directive for this session. The new framework APIs are available to it, but no template source was modified there.
